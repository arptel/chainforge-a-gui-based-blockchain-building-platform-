# ChainForge Network Simulator

## Overview

The ChainForge Network Simulator is a **live visualization layer** for real blockchain nodes generated by ChainForge. It is **not a mock simulation** — real node processes are spawned, actual transactions are submitted, and the visualizer renders real events from the running chain.

### Key Features

- **Real blockchain execution** — nodes actually run; transactions actually execute; consensus actually happens
- **Live event streaming** — WebSocket connection provides real-time events from the chain
- **9 consensus algorithms** — PoW, PoS, PoA, PBFT, Raft, Tendermint, HotStuff, Paxos, None
- **Config-driven UI** — all behavior derives from `config.yaml`; no hardcoding
- **Attack simulation** — drop nodes, partition network, slow nodes, reject blocks
- **Event replay** — record and replay transaction sequences
- **Minimal frontend** — functional HTML/JS, focus on correctness not aesthetics

## Quick Start

### Prerequisites

- Python 3.8+
- Node.js/npm (for frontend dev, optional)
- A ChainForge-generated blockchain project with `config.yaml`

### Backend Setup

1. **Install dependencies:**
   ```bash
   cd backend
   pip install -r requirements.txt
   ```

2. **Start the backend server:**
   ```bash
   python main.py /path/to/config.yaml
   ```
   
   The backend will start on `http://localhost:5000`

### Frontend

1. **Open in browser:**
   ```
   http://localhost:5000/
   ```
   Or serve `frontend/index.html` via any HTTP server:
   ```bash
   # Using Python (Unix/Mac/Windows)
   cd frontend
   python -m http.server 8080
   ```
   Then navigate to `http://localhost:8080`

## Configuration

### config.yaml Schema

All fields are required:

```yaml
network_type: permissioned        # permissioned | permissionless
governance: consortium             # consortium | democratic | single
consensus: pbft                    # pow | pos | poa | pbft | raft | tendermint | hotstuff | paxos | none
node_roles: [validator, full]      # List of allowed node roles
sync_mode: full                    # full | light | fast | snapshot | batch | realtime
max_nodes: 20                      # Maximum concurrent nodes (0 = unlimited)
block_time_ms: 3000                # Target block time in milliseconds
modules: [token, governance]       # Optional features enabled
api_endpoint: http://localhost:8545 # Optional: existing chain endpoint
require_auth: true                 # Optional: authentication required
```

## REST API Endpoints

### Health & Config

- `GET /health` — Health check
- `GET /config` — Get loaded configuration
- `POST /load-project` — Load a ChainForge project

### Node Management

- `POST /nodes` — Add a node
- `GET /nodes` — List all nodes
- `GET /nodes/<node_id>` — Get node status
- `DELETE /nodes/<node_id>` — Remove a node

### Transactions & Blocks

- `POST /transaction` — Submit a transaction
- `GET /blocks` — Get committed blocks
- `GET /blocks/<block_number>` — Get specific block

### Consensus & Metrics

- `GET /consensus-state` — Get current consensus state
- `GET /metrics` — Get network metrics

### Attack Simulation

- `POST /attack` — Execute attack (drop_node, reject_block, partition, slow_node)
- `POST /attack/stop` — Stop active attack

### Utilities

- `POST /reset` — Reset simulator to initial state
- `GET /export-trace` — Download event trace as JSON

### WebSocket

- `ws://localhost:5000/events` — Subscribe to all chain events (EVENT_TYPES, timestamps, node_id, payload)

## Event Types

All events follow this envelope on the WebSocket:

```json
{
  "type": "EVENT_TYPE",
  "timestamp": 1234567890000,
  "node_id": "node-1",
  "payload": { ... }
}
```

### Supported Events

| Event Type | Triggered By | Payload |
|---|---|---|
| `NODE_JOINED` | Node spawning | `{ node_id, role, port, block_height }` |
| `NODE_OFFLINE` | Process crash | `{ node_id, reason }` |
| `NODE_LEFT` | Graceful termination | `{ node_id }` |
| `SYNC_PROGRESS` | Node syncing | `{ current_height, target_height, progress_percent }` |
| `SYNC_COMPLETE` | Sync finished | `{ node_id, final_height }` |
| `TX_BROADCAST` | Tx submitted | `{ tx_hash, from_node, amount, memo }` |
| `CONSENSUS_PHASE` | Phase change | `{ phase, round, leader, validator_count }` |
| `VOTE_CAST` | Validator votes | `{ voter, proposer, vote='accept'\|'reject' }` |
| `BLOCK_PROPOSED` | Block proposed | `{ block_number, proposer, tx_count }` |
| `BLOCK_COMMITTED` | Block committed | `{ block_number, block_hash, proposer, votes_accepted, votes_rejected }` |
| `BLOCK_REJECTED` | Block rejected | `{ block_number, proposer, reason, votes_received }` |
| `ATTACK_ACTIVE` | Attack started | `{ attack_type, target_node_id, duration_ms }` |
| `ATTACK_STOPPED` | Attack ended | `{ attack_type }` |

## Implementation Notes

### Backend (Python)

All 9 consensus adapters (`PoWAdapter`, `PoSAdapter`, etc.) **must implement** the `IChainAdapter` interface:

```python
class IChainAdapter(ABC):
    async def spawn_node(self, role: str, port: int) -> NodeProcess
    async def terminate_node(self, node_id: str) -> None
    async def submit_transaction(self, tx: Transaction) -> str
    async def get_blocks(self, since: int) -> List[Block]
    async def get_consensus_state(self) -> ConsensusState
    def subscribe_to_events(self, handler: Callable) -> None
```

### Frontend (JavaScript)

All 9 consensus renderers (`PoWRenderer`, `PoSRenderer`, etc.) **must extend** `ConsensusRendererBase`:

```javascript
class PoWRenderer extends ConsensusRendererBase {
    renderPhase(consensusState) { /* consensus-specific visualization */ }
    renderVote(voteEvent) { /* vote visualization */ }
    renderFaultTolerance(consensusState) { /* fault tolerance info */ }
}
```

### Global State Management

The frontend maintains a global state object with four logical sections:

```javascript
{
    configSlice: { /* parsed config */ },
    networkSlice: { nodes, edges, packets, leaderNodeId },
    chainSlice: { blocks, pendingBlock, currentRound },
    uiSlice: { selectedNodeId, speedMultiplier, simulationMode }
}
```

All state updates are **driven by WebSocket events** — the frontend never fabricates state or runs timers to guess chain behavior.

## Key Constraints

1. **Real execution** — No simulation fallback. Blockchain must actually run.
2. **Event-driven UI** — All state updates from WebSocket events, not timers.
3. **Config-driven** — All behavior from config.yaml; no hardcoding.
4. **Server-side enforcement** — Permissions, limits, validations all backend.
5. **No stubs** — All 9 consensus adapters fully functional in v1.
6. **Specific errors** — "Missing config field: max_nodes" not "Invalid config".
7. **Speed slider** — Controls animation rendering only, never throttles chain.

## Troubleshooting

### Backend won't start
- Ensure `config.yaml` path is valid and accessible
- Check Python dependencies: `pip install -r requirements.txt`
- Verify port 5000 is not in use (or modify in `main.py`)

### Frontend won't connect
- Check browser console for WebSocket errors
- Verify backend is running: `curl http://localhost:5000/health`
- Check CORS settings if frontend on different port

### Nodes won't spawn
- Verify `max_nodes` not exceeded
- Check ChainForge blockchain is accessible at `api_endpoint`
- Review backend logs for process spawn errors

### No events appearing
- Check WebSocket connection in browser DevTools > Network
- Verify event bridge is subscribed to adapter
- Check backend is emitting events
