# SPEC SUMMARY

## Project Overview
The ChainForge Network Simulator is a live visualization layer for real blockchain nodes generated by the ChainForge platform. Unlike a mock simulation, it spawns actual node processes, submits real transactions, and renders animations driven entirely by live WebSocket events from the running chain. Its primary goal is to provide a visually provable demonstration that the generated chain functions correctly across various consensus and network configurations.

## Major System Components
- **Frontend (Vanilla JS):** A minimal, event-driven UI that renders node graphs, transaction lifecycles, and metrics based on real chain events.
- **Backend (Node.js/Python):** A mandatory bridge layer that spawns node processes, manages their lifecycle, proxies API calls, and streams events via WebSocket.
- **Node Process Manager:** Responsible for OS-level process management of individual blockchain nodes.
- **Event Streamer/Bridge:** Subscribes to chain node feeds, normalizes events, and pushes them to the frontend.
- **Adapter Pattern (IChainAdapter):** Modular implementation for all 9 consensus types to ensure generic backend-chain communication.
- **Config-Driven Architecture:** Entire behavior (visuals, rules, limits) is derived from `config.yaml`.

## Consensus Types (9)
| Type | Logical Requirement | Visual Requirement |
|---|---|---|
| **PoW** | Winner determined by probabilistic "hash race". | Spinning arcs around miners; "Hash Race" ticker; trophy for winner. |
| **PoS** | Selection based on stake weight. | "Validator Selection" shuffle; highlighting chosen validator. |
| **PoA** | Sequential approval by authorized nodes. | Shield icons; "Stamp" animations on block packets at each authority. |
| **PBFT** | 3-phase (Prepare, Pre-Prepare, Commit) BFT. | Mesh grid of lines for O(n²) communication; phase progress bars. |
| **Raft** | Leader-follower heartbeats; majority ACK (N/2+1). | Heartbeat pulses; election animation on leader failure. |
| **Tendermint** | 4-phase (Propose, Prevote, Precommit, Commit). | Round/height counters; 2/3+ quorum requirements display. |
| **HotStuff** | 3-phase linear pipeline. | Quorum Certificate (sealed packet) visualization; pipelined stages. |
| **Paxos** | 2-phase (Prepare/Accept) proposal numbering. | Proposal number (n) counter; acceptor state (promised/accepted) icons. |
| **None** | No consensus; unilateral commits. | Yellow warning banner; immediate slide-to-chain on commit. |

## Sync Modes (6)
| Mode | Logical Requirement | Visual Requirement |
|---|---|---|
| **Full** | Download and replay all blocks. | Solid square packets flowing; incrementing block counter. |
| **Light** | Headers only; Merkle proof verification. | Hollow square "Header" packets; "Proof Verification" step. |
| **Fast** | Snapshot + recent headers gap. | Two-stage: large snapshot packet then burst of blocks. |
| **Snapshot** | Full state at checkpoint. | Single large snapshot packet; counter jump to checkpoint. |
| **Batch** | Blocks in waves/batches. | Waves of block packets; batch progress labels. |
| **Realtime** | Joins at chain tip. | No catch-up; immediate transition to 'synced' state. |

## WebSocket Event Types (11)
1. **BLOCK_PROPOSED:** `{ blockHeight, proposerNodeId, txCount }`
2. **VOTE_CAST:** `{ voterNodeId, blockHeight, vote (accept\|reject), round }`
3. **BLOCK_COMMITTED:** `{ blockHeight, hash, proposerNodeId, txCount, commitTime }`
4. **BLOCK_REJECTED:** `{ blockHeight, reason, votesSummary }`
5. **NODE_JOINED:** `{ nodeId, role, address }`
6. **NODE_OFFLINE:** `{ nodeId, reason }`
7. **SYNC_PROGRESS:** `{ nodeId, currentHeight, targetHeight, syncMode }`
8. **SYNC_COMPLETE:** `{ nodeId, finalHeight }`
9. **CONSENSUS_PHASE:** `{ phase, round, nodeId }`
10. **TX_BROADCAST:** `{ txId, fromNodeId, toNodeId }`
11. **LEADER_ELECTED:** `{ nodeId, term/round }`

## REST Endpoints
- **Health/Config:** `GET /health`, `GET /config`, `POST /load-project`
- **Nodes:** `POST /nodes`, `GET /nodes`, `GET /nodes/<node_id>`, `DELETE /nodes/<node_id>`
- **Tx/Blocks:** `POST /transaction`, `GET /blocks`, `GET /blocks/<block_number>`
- **Consensus/Metrics:** `GET /consensus-state`, `GET /metrics`
- **Attacks:** `POST /attack`, `POST /attack/stop`
- **Utils:** `POST /reset`, `GET /export-trace`

## Server-Side Enforcement Rules
- **Node Validation:** Roles must match `config.node_roles`.
- **Permissions:** Admin-only join for permissioned networks; role-based Tx initiation.
- **Network Limits:** `max_nodes` enforced server-side.
- **Config Integrity:** Specific errors for missing mandatory fields.
- **Process Isolation:** Real process management (PID/Port tracking) enforced at backend.
